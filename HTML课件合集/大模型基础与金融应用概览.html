核心概念

大语言模型（LLM）基础

什么是大语言模型

大语言模型是基于深度学习的自然语言处理模型，通过在海量文本数据上预训练，学习语言的统计规律和语义理解能力。

Transformer架构核心组件
自注意力机制
功能: 捕捉序列内依赖关系
数学表达: \(Attention(Q,K,V) = softmax(\frac{QK^T}{\sqrt{d_k}})V\)
多头注意力
功能: 多角度理解信息
数学表达: \(MultiHead = Concat(head_1,...,head_h)W^O\)
位置编码
功能: 引入序列位置信息
数学表达: \(PE_{(pos,2i)} = sin(pos/10000^{2i/d})\)
前馈网络
功能: 非线性变换
数学表达: \(FFN(x) = max(0, xW_1+b_1)W_2+b_2\)
层归一化
功能: 稳定训练
数学表达: \(LayerNorm(x) = \gamma \frac{x-\mu}{\sigma} + \beta\)
注意力机制的直觉理解
Query (查询): "我想找什么信息？"
Key (键):     "每个位置有什么信息？"
Value (值):   "如果匹配，返回什么？"

注意力权重 = Query与所有Key的相似度（经softmax归一化）
输出 = 注意力权重加权的Value求和

class FinancialQABot:
    def __init__(self):
        self.vector_db = VectorStore()  # 向量数据库
        self.llm = ChatGPT()             # 大语言模型
        self.knowledge_base = []         # 金融知识库

    def answer(self, question):
        # 1. 检索相关知识
        relevant_docs = self.vector_db.search(question, top_k=5)

        # 2. 构建提示词
        context = "\n".join([doc.content for doc in relevant_docs])
        prompt = f"""基于以下金融知识回答用户问题。

知识库内容：
{context}

用户问题：{question}

请准确回答，如果知识库中没有相关信息，请明确说明。"""

        # 3. 生成回答
        response = self.llm.generate(prompt)

        # 4. 添加来源引用
        return self.add_citations(response, relevant_docs)

Q1: Transformer为什么比RNN更适合处理长文本？

A: 主要有三个原因：

1.并行计算
•RNN必须顺序处理，\(O(n)\)步骤
•Transformer可以并行，\(O(1)\)步骤
•GPU利用率大幅提升
2.长距离依赖
•RNN通过隐状态传递，信息逐渐衰减
•自注意力可以直接连接任意两个位置
•路径长度从\(O(n)\)降到\(O(1)\)
3.梯度传播
•RNN存在梯度消失/爆炸问题
•Transformer梯度路径更短更稳定
但也有代价
•计算复杂度\(O(n^2)\)（序列长度的平方）
•内存消耗大
•需要位置编码

Q2: 什么是大模型的"涌现能力"？为什么会出现？

A:

涌现能力（Emergent Abilities）定义

模型规模超过某个阈值后，突然获得的新能力，在较小规模时不存在。

典型涌现能力
思维链推理
出现规模: >100B
描述: 分步骤解决复杂问题
少样本学习
出现规模: >10B
描述: 几个例子就能学会新任务
代码生成
出现规模: >50B
描述: 编写正确的程序代码
数学推理
出现规模: >100B
描述: 解决数学应用题
可能的解释
1.相变理论：类似物理中的相变，量变引起质变
2.能力组合：底层能力累积到足够多，组合出新能力
3.度量问题：可能只是我们度量方式的问题
对金融应用的启示
•选择足够大的模型以获得所需能力
•较小模型可能无法完成复杂金融分析
•但更大不一定更好，要考虑成本效益

Q3: LLM的"幻觉"问题是什么？如何缓解？

A:

幻觉（Hallucination）定义

LLM生成看起来合理但实际上错误或不存在的信息。

金融场景中的幻觉风险
公司分析
幻觉表现: 编造不存在的财务数据
危害: 投资决策失误
法规咨询
幻觉表现: 引用不存在的法条
危害: 合规风险
市场预测
幻觉表现: 虚构的历史事件
危害: 错误的推断基础
缓解策略
1.检索增强生成（RAG）
•从可靠数据源检索事实
•将事实作为上下文提供给LLM
•大幅降低幻觉率
2.事实核查链
LLM生成 → 事实抽取 → 知识库验证 → 标记不一致 → 重新生成
3.置信度估计
•让模型输出置信度
•低置信度内容需要人工验证
4.温度参数调整
•降低temperature减少随机性
•牺牲创造性但提高准确性

Q8: 使用LLM进行金融分析的法律风险是什么？

A:

投资建议资质
具体表现: 无证提供投资建议
应对措施: 明确声明非投资建议
数据隐私
具体表现: 客户数据用于训练
应对措施: 本地部署、脱敏处理
知识产权
具体表现: 生成内容的版权
应对措施: 关注各国立法动态
虚假陈述
具体表现: AI幻觉导致错误
应对措施: 人工审核、免责声明
模型歧视
具体表现: 算法偏见
应对措施: 公平性审计
最佳实践
•法务部门提前介入
•建立AI使用政策
•保留决策审计轨迹
•持续关注监管动态

Q9: 开源LLM vs 闭源API，金融机构如何选择？

A:

数据安全
开源LLM: 完全可控
闭源API: 依赖供应商
定制能力
开源LLM: 可深度定制
闭源API: 有限
运维成本
开源LLM: 高（需要团队）
闭源API: 低
初始投入
开源LLM: 高
闭源API: 低
能力上限
开源LLM: 受限于自身资源
闭源API: 可获得最强模型
合规风险
开源LLM: 可控
闭源API: 需评估
决策建议
大型银行/券商
推荐选择: 开源+自研
原因: 数据敏感、有资源
中型机构
推荐选择: 混合方案
原因: 平衡成本与安全
小型机构/初创
推荐选择: 闭源API
原因: 快速上线、低成本
